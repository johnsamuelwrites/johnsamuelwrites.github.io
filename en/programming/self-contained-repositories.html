During past several weeks, I have been analysing several open-source projects especially on Github1 and in particular their (code) repositories. What surprises me is that majority of the projects are not self-contained. What do I mean by self-contained repositories? Everything concerning the project is available in one single place, i.e., within the project repository itself. In most of the projects, the entire code is on one website, the documentation on another web site, any application programming interface (API) description on a third website, continuous integration details on yet another website. If one has to request for new features or report any bugs, they are once again on one another website. Then there are discussions around a new feature or a bug that can only be found in some mailing lists. In short, one has to search multiple places just to understand what's going on, what the project is all about or why a particular decision was made. Another major problem is Link rot2

Checklist

A self-contained repository is a repository where you can find everything concerning the tool or application: all discussions, complete documentation both for end-users and developers, screenshots,...

What are the contents of a self-contained repository? or to put in other words, what is the checklist for a self-contained repository?

  • Source code
    • Main source code
    • Debugging options
    • Logging options
    • dependency (package)
    • Unit tests, Integration tests, functional tests, stress tests, performance tests
  • Specifications
    • Requirement analysis (platform requirements, hardware, software, memory)
    • Design: UML diagrams, flow charts, data flow diagrams
    • verification and validation
  • Logo
    • different sizes (or svg)
  • User Interface
    • Feature Requests
    • Discussions and Resolution
  • Application Programming Interface (API)
    • various programming languages
    • REST or other web service API architecture
  • Todo List
    • Feature Requests (e.g. user interface mockups)
    • Discussions and Resolution
    • Status
  • Issues List
    • Bugs
    • Discussions and Resolution
    • Status
  • Discussions
    • Meetings, discussions, and resolutions
  • Code Review
    • code reviews, discussions,
  • Packaging
    • packaging - packaging for various os or distibutions
    • containers (docker containers, linux containers or open containers)
  • Version Control
    • distributed like git
    • Files to be ignored from the repositories like .gitignore
  • Profile
    • development details: continuous integration, code coverage, system and memory profile
  • Releases
    • Feature releases, issues resolved
    • dependency (package)
  • Verification and Validation
    • Test case
  • Documentation (wiki and manpages)
    • quick start, compilation
    • application/program description
    • application compilation, installation, development and testing
    • UI Screenshots
    • documentation, command line options, gui, man pages
The above is not an exhaustive list.
